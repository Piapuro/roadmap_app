🧙 要件定義ウィザード（新規作成）

チームが開発するプロダクトの要件を5つのステップで入力するウィザード画面を作成しました。 STEP1〜5の入力を Zustand で一元管理し、最終ステップで「AI分析を開始」ボタンを押すと バックエンドAPIに要件データを送信してロードマップ生成を開始します。

GET /requirements/new

📄 要件定義ビュー（新規作成）

生成済みの要件定義を閲覧するページを作成しました。 プロダクト種別・人数・期間・経験レベルのサマリーバー、機能一覧、 AI生成ステータス、AIレビュー結果を表示します。 「AIで再生成」「要件を編集」ボタンも配置しています。

型定義 src/types/requirements.ts

要件定義機能全体で使う TypeScript の型とラベル定数 をまとめたファイルです。

  • ProductType — プロダクト種別の型。'web' | 'app' | 'game' | 'ai' の4択。
  • ExperienceLevel — チーム経験レベルの型。'beginner' | 'intermediate' | 'advanced' の3択。
  • RequirementStatus — 要件のステータス。'draft'（下書き）| 'locked'（確定済み）。
  • WizardState — ウィザードの入力内容をすべて格納する型（STEP1〜5の全フィールド + 現在のステップ番号 + 送信中フラグ）。
  • Requirement — APIから返ってくる要件定義データの型（id, teamId, features配列など）。
  • PRODUCT_TYPE_LABELS — プロダクト種別を日本語に変換するマップ（例: 'web' → 'Webアプリ'）。
  • EXPERIENCE_LEVEL_LABELS — 経験レベルを日本語に変換するマップ（例: 'beginner' → '初心者中心'）。
  • REQ_WIZARD_STEPS — 5ステップの定義配列。ステッパーとサイドバーの両方で使う表示ラベルを持つ。
Record<ProductType, string> を使うことで、ProductType に新しい値を追加し忘れた場合にコンパイルエラーで気づけるようになっています。
定数 src/client/constants/featureList.ts

STEP2「機能・要件選択」で表示する 機能一覧のデータ定義 ファイルです。

  • 型 FeatureCategory で「カテゴリID / カテゴリ名 / 機能名リスト」のまとまりを定義。
  • FEATURE_LIST は Record<ProductType, FeatureCategory[]> 型で、プロダクト種別ごとに異なる機能カテゴリを持つ。
  • 例：web なら「認証・ユーザー管理」「チーム・コラボ」「決済・外部連携」「コンテンツ・UI」の4カテゴリ。
  • 例：ai なら「UI・入出力」「AI連携」「データ管理」の3カテゴリ。
プロダクト種別を変えると表示される機能リストも変わります。ここを編集するだけで機能の追加・削除ができます。
ストア src/client/store/wizardStore.ts

ウィザードの入力状態をすべて管理する Zustand ストア です。

  • ページをまたがずに状態を保持し、STEP1〜5 の全コンポーネントから同じ状態を読み書きできます。
  • setProductType(v) — プロダクト種別を更新。同時に selectedFeatures をリセットして、前の種別の選択が残らないようにします。
  • toggleFeature(name) — 機能名を渡すと選択追加/解除をトグルします。
  • setTeamScale({ memberCount, periodWeeks, experienceLevel }) — STEP3 の3つの値をまとめて更新。
  • setFreeText(text) / setSupplementUrl(url) — STEP4 の2フィールドを個別に更新（まとめると高速タイピング時に古い値が混入するため分離）。
  • nextStep() / prevStep() / goToStep(n) — ステップ移動。
  • reset() — ウィザード完了後やキャンセル時に全状態を初期値に戻す。
フック src/client/hooks/useWizard.ts

ステップごとのバリデーション処理をまとめた カスタムフック です。

  • validateStep(step) を呼ぶと { valid: boolean, errors: Record<string, string> } を返します。
  • STEP1 — プロダクト種別が未選択なら NG。
  • STEP2 — 機能が1件も選択されていなければ NG。
  • STEP3 — 人数（1〜20の整数）・期間（1〜52週の整数）・経験レベルの3項目を検証。
  • STEP4 — 補足メモが1000文字超、または参考URLが https:// 以外の形式なら NG。
整数チェックは「整数かどうか」を先に確認してから範囲チェックをする順番になっています（逆順だとデッドコードになるため）。
コンポーネント src/client/components/requirements/WizardShell.tsx

ウィザード画面全体の 骨格コンポーネント です。左サイドバー・上部ステッパー・中央コンテンツ・下部ナビゲーションフッターを組み合わせています。

  • ヘッダー — 現在のステップタイトルと説明文を表示。STEP_CONFIG 配列からステップ番号に応じた内容を出す。
  • コンテンツ切り替え — currentStep の値に応じて Step1〜Step5 のどれかを表示する（ページ遷移なし）。
  • フッター（次へ / 前へ） — 「次へ」を押すと validateStep() でバリデーション。失敗ならエラー表示、成功なら次ステップへ進む。STEP5 の「AI分析を開始」押下時だけ API 送信処理を行う。
  • API送信処理 — saveRequirement()（Server Action）を呼び、成功したら /ai-loading にリダイレクト。失敗時は isSubmitting を解除してリトライ可能にする。
  • teamId — 現在は localStorage.getItem('team_id') から取得（#006 マージ後にセッションへ変更予定）。空の場合はエラーを表示して送信を止める。

コンポーネント src/client/components/requirements/Step1ProductType.tsx

STEP1「プロダクト種別を選択」の 入力画面 です。

/requirements/[reqId] で生成済み要件を閲覧する ビューコンポーネント です。APIから取得した Requirement オブジェクトを受け取って表示します。

  • ヘッダー — 「チーム > 要件定義」のパンくず + 「AIで再生成」「要件を編集」ボタン。要件が locked（確定済み）の場合は「要件を編集」が非活性になる。

Server Action src/server/actions/requirements.ts

バックエンドAPIとの通信処理を担う Next.js Server Actions ファイルです（'use server' 宣言）。クライアントから直接呼び出せます。

  • saveRequirement(input) — STEP5 の「AI分析を開始」で呼ばれます。POST /api/v1/teams/{teamId}/requirements にリクエストを送り、レスポンスの Requirement オブジェクトを返す。
  • getRequirement(teamId, reqId) — /requirements/[reqId] ページのサーバー側で呼ばれます。GET /api/v1/teams/{teamId}/requirements/{reqId} で要件を取得して返す。
  • 両関数とも API_URL 環境変数が未設定の場合は早期エラーを投げる。res.ok チェックと data?.id の最低限バリデーションも実施。
  • URLにパスパラメータを含める際は encodeURIComponent を適用してURLインジェクションを防ぐ。

ページ src/app/(dashboard)/requirements/[reqId]/page.tsx

/requirements/[reqId] のルートに対応する Server Componentページ です。

  • URLの [reqId] パラメータと ?teamId= クエリを受け取り、サーバー側で getRequirement() を呼んでデータを取得。
  • 取得に失敗した場合（APIエラーや存在しない reqId など）は notFound() を呼んで 404 ページを表示。
  • 取得成功時は <RequirementsView requirement={...} /> に渡して表示。
現在 teamId をクエリパラメータから取得しているため、他チームのIDを指定すればデータを取得できてしまいます（IDOR）。#006（Supabase Auth）マージ後にサーバーサイドセッションから取得するよう変更予定です。
