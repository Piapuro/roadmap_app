### [C-2] `src/client/components/onboarding/SkillsForm.tsx:47-51` — スキップ時のデータ整合性リスク

スキップボタンは検証・保存処理を一切行わずに `/dashboard` へ遷移する。この状態では `users.skill_level` が `NULL` のまま残る可能性がある。

```typescript
// src/client/components/onboarding/SkillsForm.tsx:172-178
<button
  type="button"
  onClick={() => router.push("/dashboard")}
  className="text-sm text-muted-foreground ..."
>
  スキップ
</button>
```

スキップ後にダッシュボードや AI 機能が `skill_level` を参照する際、`NULL` を想定していないコードであれば実行時エラーになる。少なくともデフォルト値を書き込む、または DB スキーマ側で `DEFAULT 'beginner'` を設定すべきである。

**修正案（クライアントでデフォルト値を保存してからスキップ）:**

```typescript
async function handleSkip() {
  const supabase = createClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (user) {
    await supabase
      .from("users")
      .update({ skill_level: "beginner", onboarding_skipped: true })
      .eq("id", user.id);
  }
  router.push("/dashboard");
}

// JSX
<button type="button" onClick={handleSkip} ...>
  スキップ
</button>
```

### [C-3] `src/client/components/auth/RegisterForm.tsx:70-91` — エラー後の `isPending` リセット漏れ

`supabase.auth.signUp()` がエラーなく成功し、かつ `data.session` が存在する場合（メール確認不要の設定）、`router.push("/onboarding")` が呼ばれるが `isPending` が `false` にリセットされない。遷移に失敗した場合やブラウザバックしたとき、ボタンが永続的に無効化されたままになる。

```typescript
// src/client/components/auth/RegisterForm.tsx:85-90
if (data.session) {
  router.refresh();
  router.push("/onboarding");
  // isPending が true のまま残る
} else {
  router.push("/register/check-email");
  // こちらも isPending が true のまま残る
}
```

**修正案: `finally` でリセットを保証**

```typescript
async function handleSubmit(e: React.FormEvent) {
  e.preventDefault();
  const next = validate();
  if (Object.keys(next).length > 0) {
    setErrors(next);
    return;
  }
  setErrors({});
  setIsPending(true);
  try {
    const supabase = createClient();
    const { data, error } = await supabase.auth.signUp({ ... });
    if (error) {
      setErrors({ general: toJaErrorMessage(error.message) });
      return;
    }
    if (data.session) {
      router.refresh();
      router.push("/onboarding");
    } else {
      router.push("/register/check-email");
    }
  } finally {
    setIsPending(false);
  }
}
```

## Major Issues

### [M-1] `src/client/components/onboarding/SkillsForm.tsx:47-61` — クライアントから直接 DB 書き込み（アーキテクチャ・セキュリティ懸念）

`users` テーブルへの `UPDATE` と `user_skills` テーブルへの `INSERT` をブラウザクライアント（`createBrowserClient`）から直接実行している。これは Supabase の Row Level Security (RLS) が正しく設定されていれば機能するが、RLS の設定ミスがあった場合に他ユーザーのデータを上書きできる脆弱性になる。

```typescript
// src/client/components/onboarding/SkillsForm.tsx:47-61
const { error: updateError } = await supabase
  .from("users")
  .update({ skill_level: skillLevel })
  .eq("id", user.id); // user.id はクライアント側で取得した値
```

プロジェクトの規約（`src/server/` = Server-side only）に照らすと、DB 書き込みは Server Action または API Route 経由にすべきである。

**推奨修正案: Server Action を使用**

```typescript
// src/server/actions/onboarding.ts
"use server";
import { createClient } from "@/server/lib/supabase-server";
import type { SkillLevel } from "@/types/onboarding";

export async function saveOnboardingSkills(
  skillLevel: SkillLevel,
  skillIds: string[]
) {
  const supabase = await createClient();
  const { data: { user }, error: userError } = await supabase.auth.getUser();
  if (userError || !user) throw new Error("認証が必要です");

  const { error: updateError } = await supabase
    .from("users")
    .update({ skill_level: skillLevel })
    .eq("id", user.id);
  if (updateError) throw new Error("スキルレベルの保存に失敗しました");

  // user_skills の INSERT は変わらず
}
```

### [M-2] `src/client/components/onboarding/SkillsForm.tsx:53-61` — `user_skills` の重複 INSERT に対する考慮なし

`user_skills` テーブルへの `INSERT` 時、同一ユーザーが `/onboarding` を再訪問した場合（ブラウザバック等）に重複行が挿入される。`UNIQUE` 制約や `upsert` が使われていない。

```typescript
// src/client/components/onboarding/SkillsForm.tsx:54-62
const skillRows = SKILL_TAGS.filter((t) => selectedTags.has(t.id)).map((t) => ({
  user_id: user.id,
  skill_name: t.name,
  is_learning_goal: false,
}));
const { error: skillsError } = await supabase.from("user_skills").insert(skillRows);
```

**修正案: `upsert` を使用し冪等性を確保**

```typescript
const { error: skillsError } = await supabase
  .from("user_skills")
  .upsert(skillRows, { onConflict: "user_id,skill_name" });
```

### [M-3] `src/client/components/auth/RegisterForm.tsx:227-229` — エラーメッセージとバリデーションの矛盾

フロントエンドのバリデーション（50行目）では「パスワードは8文字以上」と表示されるが、Supabase エラーメッセージの日本語変換（229行目）では「パスワードは6文字以上」と表示される。

```typescript
// src/client/components/auth/RegisterForm.tsx:48-49（バリデーション）
} else if (form.password.length < 8) {
  next.password = "パスワードは8文字以上で入力してください";

// src/client/components/auth/RegisterForm.tsx:229（エラー変換）
if (msg.includes("Password should be"))
```

ユーザーは「8文字入力したのになぜ6文字以上と言われるのか」と混乱する。Supabase 側のデフォルト最低文字数は6文字であるが、フロントで8文字以上を要求しているなら、Supabase Dashboard でも同様に設定するか、エラーメッセージを統一すべきである。

**修正案:**

```typescript
if (msg.includes("Password should be"))
  return "パスワードは8文字以上で入力してください";
```

### [S-1] `src/client/components/onboarding/SkillsForm.tsx` — 選択タグ数の上限設定

技術タグに選択数の上限を設けることで、AI のスキル分析精度向上と UX 改善が期待できる。例えば最大 5〜10 個に制限し、上限到達時はメッセージを表示する。

```typescript
const MAX_TAGS = 10;

function toggleTag(tagId: string) {
  setSelectedTags((prev) => {
    const next = new Set(prev);
    if (next.has(tagId)) {
      next.delete(tagId);
    } else if (next.size < MAX_TAGS) {
      next.add(tagId);
    }
    return next;
  });
}
```

## Summary

### 総合評価

**スコア: 3.5 / 5.0**

UI 実装の完成度は高く、型安全性や React の状態管理パターンは適切。しかし、認証・認可周りに複数の問題がある。

### 主要所見

1. **最重要**: `/onboarding` ページが未認証アクセスを許可している（`proxy.ts` の `PROTECTED_PATHS` への追加と Server Component での `getUser()` チェックが必要）。

2. **データ整合性**: スキップ機能が `skill_level = NULL` を残す可能性があり、ダウンストリームの機能に影響する。また `user_skills` の重複 INSERT に対する冪等性処理がない。

3. **アーキテクチャ**: プロジェクト規約（`src/server/` = Server-side only）に沿って、DB 書き込みを Server Action 化することを推奨する。

4. **UX 矛盾**: パスワード最小文字数のメッセージが フロント（8文字）と Supabase エラー変換（6文字）で食い違っている。

5. **エラーハンドリング**: `handleSubmit` で `isPending` が `finally` でリセットされないため、遷移後のブラウザバック時にフォームが操作不能になる可能性がある。

Critical Issues の修正（C-1〜C-3）は本ブランチのマージ前に対応必須。Major Issues（M-1〜M-4）はスプリント内での対応を推奨する。
