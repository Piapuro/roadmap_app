## 🔴 Critical Issues

### [C-1] 保存成功後に再送信が失敗すると `success` と `error` が同時表示される

**ファイル**: `src/client/components/team/TeamSettingsForm.tsx:67`
**問題**: `handleSubmit` 内で `setError(null)` はリセットされるが、`setSuccess(false)` が呼ばれていない。1回目の保存が成功（`success=true`）した後、2回目の保存がAPIエラーで失敗した場合、「チーム設定を保存しました」（success）と「更新に失敗しました」（error）の両バナーが同時に表示される矛盾した状態になる。

```tsx
// ❌ 現在のコード
setError(null)
startTransition(async () => {
  try {
    await updateTeamSettings({ ... })
    setSuccess(true)
  } catch (err) {
    setError(err instanceof Error ? err.message : 'チーム設定の更新に失敗しました')
  }
})
```

```tsx
// ✅ 修正案: startTransition前にsuccessもリセット
setError(null)
setSuccess(false)   // ← 追加
startTransition(async () => {
  try {
    await updateTeamSettings({ ... })
    setSuccess(true)
  } catch (err) {
    setError(err instanceof Error ? err.message : 'チーム設定の更新に失敗しました')
  }
})
```

**理由**: 「保存済み」と「エラー」が同時に表示されると、ユーザーが現在の保存状態を誤認する。

**ファイル**: `src/server/actions/team.ts:20–21`
**問題**: `getTeamSettings` / `updateTeamSettings` では `satisfies` キーワードで型チェックしているが、`getMembers` だけ `(data as any[]).map(d => ({...}))` という別パターンを使用。プロジェクト内で同じ目的のコードが2つのパターンに分裂している。

```ts
  name:     d.name,

```

```ts
// ✅ 修正案: settingsと同一パターンに統一
const data: any[] = await res.json()
return data.map(d => ({
  id:       d.id       as string,
  name:     d.name     as string,
  email:    d.email    as string,
  role:     d.role     as Role,
  joinedAt: d.joined_at ?? d.joinedAt ?? '',
} satisfies TeamMember))
```

**理由**: コードベース内でパターンが統一されていないと、後からメンテする人がどちらが正しいか迷う。`satisfies` を使うとコンパイル時に `TeamMember` の構造に合致しているか検証される。
